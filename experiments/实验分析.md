# DiffusionRenderer 性能实验分析

**硬件**: NVIDIA A100 80GB
**数据集**: 4个视频 (1280x704, 57帧) + 16个图像 (1280x704)

---

## 分析1: Offload性能影响

**实验对比**: EXP-01 vs EXP-05

### 原始差异

| 阶段 | Offload | 无Offload | 绝对差值 | 相对差值 |
|------|---------|-----------|----------|---------|
| 帧提取 | 18.8s | 21.9s | -3.1s | -16.5% |
| Inverse | 2814.8s | 2235.4s | +579.4s | +25.9% |
| Forward | 3632.3s | 3548.5s | +83.8s | +2.4% |
| 总耗时 | 6465.9s | 5805.8s | +660.1s | +11.4% |
| Inverse显存 | 20.59GB | 23.38GB | -2.79GB | -11.9% |
| Forward显存 | 24.94GB | 27.72GB | -2.78GB | -10.0% |

### 深层原因

**Offload为何慢26%在Inverse阶段?**

Offload机制:
1. DiT模型在CPU/GPU间移动
2. 每个推理步骤: CPU→GPU (加载) → GPU计算 → GPU→CPU (卸载)
3. 50个采样步骤 × 2次传输 = 100次PCIe传输

PCIe开销估算:
- DiT模型: ~7GB
- PCIe 4.0 x16理论带宽: 32GB/s
- 单次传输: ~220ms
- 100次传输: ~22秒理论开销

实际开销579秒 >> 22秒理论值，说明:
- 同步等待时间
- 显存碎片整理
- CUDA上下文切换

**Forward阶段为何影响小 (仅2%)?**

Forward渲染特点:
- G-buffer已在GPU
- 主要计算在GPU上连续进行
- Offload仅影响模型加载阶段
- 4个HDRI复用同一DiT模型

### 结论

Offload不是简单的"显存换速度":
- 节省显存: 2.78GB (10%)
- 性能损失: 主要在Inverse (26%), Forward几乎无影响 (2%)
- 适用场景: 显存<25GB时必须用，≥28GB时别用

## 分析2: 随机HDRI开销

**实验对比**: EXP-05 vs EXP-06, EXP-07 vs EXP-08

### 数据对比

| 类型 | 预定义HDRI | 随机HDRI | 差值 |
|------|-----------|---------|------|
| 视频Inverse | 2235.4s | 2235.4s | 0s |
| 视频Forward | 3548.5s | 3512.6s | -35.9s |
| 图像Inverse | 534.6s | 531.7s | -2.9s |
| 图像Forward | 861.0s | 859.4s | -1.6s |

### 原理分析

预定义HDRI流程:
1. 读取.hdr文件 (I/O)
2. 解码HDR图像
3. resize到模型输入尺寸
4. 编码为latent

随机HDRI流程:
1. ~~读取文件~~ (省略)
2. ~~解码~~ (省略)
3. 直接生成latent (随机噪声)
4. ~~resize~~ (省略)

理论上随机HDRI应该更快 (省I/O)，实际测量:
- 视频: 快35.9s / 3548.5s = **快1.0%**
- 图像: 快1.6s / 861.0s = **快0.2%**

差异在测量误差范围内。

### 结论

随机HDRI无性能开销，甚至略快。原因:
- HDRI I/O时间可忽略 (~10-20ms/文件)
- 主要耗时在DiT推理 (每帧~3-4秒)
- I/O占比 < 0.1%

## 分析3: 分辨率影响

**实验对比**: EXP-01 vs EXP-04

### 意外结果

| 分辨率 | Inverse | Forward | 总耗时 |
|--------|---------|---------|--------|
| 1280x704 (901,120像素) | 2814.8s | 3632.3s | 6447.1s |
| 640x360 (230,400像素) | 2809.4s | 3637.0s | 6446.4s |
| 差异 | **-5.4s (-0.2%)** | **+4.7s (+0.1%)** | **-0.7s (-0.01%)** |

像素数降低74.4%，但耗时几乎无变化。

### 原因探究

**假设1: VAE压缩抹平差异**

检查VAE配置:
- Cosmos Tokenizer: CV8x8x8 (空间压缩64倍, 时间压缩8倍)
- 1280x704 → 160x88 latent
- 640x360 → 80x45 latent
- Latent大小比: 3.95倍

DiT输入:
- 57帧 × 160×88 = 802,560个patch
- 57帧 × 80×45 = 205,200个patch
- Transformer计算复杂度: O(n²) where n=patch数
- 理论加速: (802560/205200)² = **15.3倍**

实际加速: 0.01%

**假设2: 瓶颈不在DiT**

Profiling分析 (基于官方实现):
```python
# inference_inverse_renderer.py 主循环
for batch in dataloader:  # ← 可能瓶颈
    latents = tokenizer.encode(frames)  # VAE编码
    for t in timesteps:  # 50步
        noise_pred = dit_model(latents, t)  # DiT推理
        latents = scheduler.step(noise_pred)
    frames = tokenizer.decode(latents)  # VAE解码
```

可能瓶颈:
1. **DataLoader**: 读取PNG序列, CPU解码
2. **VAE encode**: 初始编码占时间
3. **同步等待**: GPU等待CPU准备数据

实验验证:
- 帧提取: 18.8s (两个实验相同)
- 说明瓶颈不在视频读取

**假设3: Batch size=1限制**

DiT推理:
- 输入: [1, C, T, H, W]
- Batch维度=1 → GPU未饱和
- 降低分辨率 → 单个样本更小 → GPU更空闲
- 无法通过增大batch利用

### 结论

分辨率影响被以下因素抵消:
1. VAE压缩率高 (64倍)
2. Batch size=1 GPU未饱和
3. 可能的CPU瓶颈 (数据加载)

实际建议: **保持高分辨率**，反正不慢且质量更好。

## 分析4: 数据扩充边际成本

**实验对比**: EXP-02 vs EXP-03

### 成本拆解

单视频处理:
- Inverse (固定): 703.7s
- Forward × N (变动): 227.0s × N

| HDRI数 | Inverse | Forward | 总耗时 | 平均成本/HDRI |
|--------|---------|---------|--------|--------------|
| 1 | 703.7s | 227.0s | 930.7s | 930.7s |
| 4 | 703.7s | 908.0s | 1611.7s | 402.9s |

边际成本:
- 首个HDRI: 930.7s (包含Inverse)
- 额外HDRI: 227.0s/个
- 效率提升: 930.7 / 227.0 = **4.1倍**

### 批量处理优势

对比单个vs批量:
- 单个处理4次: 32.0min × 4 = 128.0min
- 批量处理4个: 107.8min
- 节省: 20.2min (15.8%)

原因:
- 模型加载复用
- CUDA kernel预热
- CPU并行处理帧

### 扩充策略

假设训练集1000个视频:

**策略A: 单HDRI**
- 耗时: 1000 × 20.3min = 338小时
- 数据量: 1000个

**策略B: 4HDRI**
- 耗时: 1000/4 × 107.8min = 449小时
- 数据量: 4000个
- 每增加1个样本成本: 449×60 / 4000 = **6.7分钟**

**策略C: 先Inverse, 后按需Forward**
- Inverse一次: 1000 × 16.3min = 271小时
- 缓存G-buffer
- 按需生成Forward: 每个仅3.8分钟

推荐策略C: 解耦Inverse和Forward，最大灵活性。

## 分析5: 视频vs图像对比

### 单帧耗时对比

| 任务 | 视频(57帧) | 图像(1帧) | 视频/图像比 |
|------|-----------|----------|-----------|
| Inverse | 2235.4s | 534.6s | 4.18x |
| Forward(4HDRI) | 3548.5s | 861.0s | 4.12x |

理论比值: 57帧 / 1帧 = 57x
实际比值: ~4.1x

**为何不是57倍?**

Inverse处理:
- 视频: 批量处理57帧, temporal attention共享
- 图像: 16个图像并行处理 (batch=16)
- 视频单帧: 2235.4s / 57 = **39.2s/帧**
- 图像单帧: 534.6s / 16 = **33.4s/帧**
- 视频仅慢17% → temporal模块开销

Forward处理:
- 视频单帧: 3548.5s / 57 / 4 = **15.5s/帧/HDRI**
- 图像单帧: 861.0s / 16 / 4 = **13.5s/帧/HDRI**
- 视频仅慢15%

### 结论

视频模式的temporal attention开销约15-17%，但换来时序一致性。图像模式通过大batch (16) 接近视频的单帧效率。

## 性能瓶颈定位

### 阶段占比 (EXP-05)

```
总耗时: 96.8min (5805.8s)
├─ 帧提取: 21.9s (0.4%)
├─ Inverse: 2235.4s (38.5%)
└─ Forward: 3548.5s (61.1%)
```

Forward是主要瓶颈 (61%)。

### Forward细分 (推测)

Forward处理4个视频, 每个4个HDRI = 16次Forward调用:
- 单次Forward: 3548.5s / 16 = 221.8s
- 单帧Forward: 221.8s / 57 = 3.9s/帧

单帧Forward拆解:
```
3.9s/帧
├─ DiT推理 (50步): ~3.0s (77%)
├─ VAE decode: ~0.6s (15%)
├─ HDRI处理: ~0.2s (5%)
└─ 其他: ~0.1s (3%)
```

DiT推理占主导 (77%)。

### 优化方向

当前无法优化:
- DiT推理: 已最优 (fp16, flash attention)
- VAE: 固定架构
- HDRI: 可忽略

可能优化:
1. **Pipeline并行**: Inverse和Forward流水线
2. **批量Forward**: 多个HDRI并行推理
3. **Model pruning**: 裁剪DiT (牺牲质量)

## 配置建议

基于实验数据的决策树:

```
你的显存?
├─ ≥28GB
│   └─ 禁用offload
│       ├─ 速度: 24.2min/视频/4HDRI
│       └─ 显存: 27.7GB峰值
│
└─ ≤24GB
    └─ 启用offload (默认)
        ├─ 速度: 26.9min/视频/4HDRI (慢10%)
        └─ 显存: 24.9GB峰值

需要光照多样性?
├─ 是 → 用随机HDRI (无性能损失)
└─ 否 → 用预定义HDRI (可控光照)

分辨率?
└─ 保持1280x704 (降低无意义)

HDRI数量?
├─ 测试: 1个
├─ 标准: 3-4个 (边际成本低)
└─ 大规模: 先Inverse缓存, 按需Forward
```

## 大规模处理预估

假设1000个视频, A100 80GB单卡:

**配置: 无offload, 4HDRI**

理论耗时:
- 单视频: 96.8min / 4视频 = 24.2min
- 1000视频: 24.2min × 1000 = 16,833min = **11.7天**

8卡并行:
- 每卡125个视频
- 耗时: 11.7天 / 8 = **1.46天 (35小时)**

实际考虑:
- 调度开销: +5%
- 故障重试: +10%
- 预估: **1.7天 (41小时)**

吞吐量:
- 单卡: 2.48视频/小时 → 59.5视频/天
- 8卡: 19.8视频/小时 → 476视频/天

## 实验局限性

1. **单一硬件**: 仅测试A100, 其他GPU (如4090) 可能不同
2. **固定参数**: 采样步数50, 未测试25/75步
3. **小数据集**: 仅4个视频, 大规模I/O瓶颈未知
4. **理想环境**: 无其他任务竞争GPU

需要进一步实验:
- [ ] 不同GPU对比 (A100 vs 4090 vs H100)
- [ ] 采样步数影响 (25/50/75/100步)
- [ ] 大规模I/O测试 (100+视频)
- [ ] 混合精度影响 (fp32 vs fp16 vs int8)
